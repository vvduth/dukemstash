## Key findings from first audit (2026-03-27)

### Critical
- `updateItem` in `src/lib/db/items.ts` (line 198): `tagsOnItems.deleteMany` runs with only `{ itemId }` — no `userId` check. A race window exists where tags are deleted before the `item.update` ownership check fires. An attacker who can supply an arbitrary `itemId` to the server action could strip tags from another user's item. Mitigation: wrap both operations in a `$transaction` or move the `deleteMany` after a prior ownership check.
- `/api/files/[key]` route: authenticates the user but does NOT verify the requested R2 key belongs to that user. Any authenticated user can fetch any other user's file by guessing the key format (`userId/uuid.ext`). Keys are stored in `item.fileUrl`. The fix is to look up the item by `fileUrl` and confirm `userId` matches before streaming the file.

### High
- `ItemCard`, `ImageCard`, `FileListRow` — all use `useState` and browser APIs (`navigator.clipboard`) but are missing `'use client'` directive. They work because they're only ever rendered inside the `ItemGridWithDrawer` client component, but this is fragile — if they were ever imported by a server component it would fail.
- Dashboard layout + page + items type page all do a redundant DB lookup: they call `auth()` to get a session with `session.user.id` populated, then immediately do another `prisma.user.findUnique({ where: { email } })` to get the user ID. The session JWT already contains `token.sub` = user.id (set in `auth.ts` jwt callback). `session.user.id` is available directly — no extra DB hit needed.
- `src/lib/mock-data.ts` is no longer used anywhere in the application (all pages use real DB queries). Unused file left in codebase.

### Medium
- `updateItem` tag delete/reconnect is not atomic: `deleteMany` then `update` are two separate DB calls. If the `update` fails (e.g. wrong userId), the tags are already gone. Should be wrapped in `$transaction`.
- `getItemsByType` has no `take` limit. A user with thousands of items will load them all in one query — no pagination.
- `getRecentItems` and `getPinnedItems` share a near-identical result shape and mapping function (duplicated ~20 lines each). Could be a shared mapper.
- `deleteItem` does a `findFirst` then `delete` in two queries — could be one `deleteMany` with a `{ id, userId }` where clause.

### Architecture notes
- Auth session does have `user.id` via JWT callback in `auth.ts` (lines 14-19). The redundant user lookups in the layout and page files are unnecessary DB hits.
- Rate limiting is IP-based, fail-open (correct design decision).
- File validation is both client-side and server-side (correct).
- All server actions check `session.user.id` (correct ownership pattern).
- `itemTypeId` passed to `createItem` is not validated to be a real system type ID. A user could pass any string, and Prisma would throw a foreign key error (caught generically). Low risk since auth is required, but error message is opaque.
